### 2026-06-02 — Session 44: Widescreen cutscene framing — 4:3-authored shots hold their frame (fixes #36)

- **`fix(cutscene): clamp in-engine cutscene cameras to 4:3 horizontal framing on widescreen`** — cutscene cameras inherit the window aspect (`ONcMotoko_AspectRatio` = `M3rDraw_GetWidth()/GetHeight()`), so on a 16:9 display the horizontal FOV widened 57.8°→72.7°, opening the left/right view-frustum planes ([`Motoko_Geom_Camera.c:166`](BungieFrameWork/BFW_Source/BFW_Motoko/Manager/Motoko_Geom_Camera.c) `fovx02 = atan(aspect·tan(fovy/2))`) that feed **both** the visibility cull and the GL projection. That pulled a near wall the 4:3-authored level-2 intro shot framed *out* into view — ~50u from camera, dominating the frame. Fix: in the per-frame render setup ([Oni_GameState.c](OniProj/OniGameSource/Oni_GameState.c), just before `AKrEnvironment_StartFrame`), when `local.in_cutscene` and the window is wider than 4:3, lower the camera's vertical FOV so the *horizontal* FOV equals 4:3's — `fovy' = 2·atan((4:3 ÷ aspect)·tan(fovy/2))`, new constant `ONcMotoko_CutsceneAspect` ([Oni_Motoko.h](OniProj/OniGameSource/Oni_Motoko.h)) — filling the wide screen by trimming vertical instead of widening horizontal (no pillarbox). Sourced from the authored global FOV so it never telescopes; outside cutscenes it re-asserts the authored FOV, auto-restoring on cutscene exit. General to every in-engine cutscene, not an L2 patch. **Port-exposed**, not a Bungie bug: the aspect math is correct for 4:3; the modern-Mac widescreen default exposes it. The long elimination chain that preceded the fix — camera position/orientation proven == authored `Camout01`, `env_show` flag mechanism + remap byte-identical to OniSplit ground truth, culling code byte-identical to Bungie 2000-12-21 — is in [#36](https://github.com/andiyar/OniARM64/issues/36); the 4:3 A/B test isolated aspect as the sole causal variable. User-verified at 1920×1080: wall gone, establishing shot framed cleanly, nothing clipped. Fixes [#36](https://github.com/andiyar/OniARM64/issues/36).
- **`release: 1.3.0r1`** ([Info.plist](macos/Info.plist)) — consolidated the private prerelease line onto a single clean **1.3.0** release and dropped the stray **1.3.1** tags (the repo is private and none had been announced, so the older DMGs were deleted rather than renumbered). 1.3.0r1 supersedes every earlier private build: it carries the Apple Silicon port (levels 1–4), Mac + PC `GameDataFolder` drop-and-play, native AVFoundation intro/outro cinematics, and the widescreen cutscene-framing fix above. Kept **1.3.0a1** as the historical first-alpha marker; future releases increment the R number (1.3.0r2, …). Built via `make oni_app_release` (the #36 investigation diagnostics stashed during the build so the public binary ships clean), signed + notarized + stapled, published to [Releases](https://github.com/andiyar/OniARM64/releases) as a prerelease.

### 2026-06-01 — Session 43: Mac retail SNDD loads + plays natively (addresses #37)

First half of Mac retail `GameDataFolder` drop-and-play — the original 2001 Mac disc data now loads and its sound plays on the ARM64 build, no PC data required. Two paired fixes (the crash fix exposed the next layer — the classic cascade):

- **`feat(mac-data): Mac SNDD layout — engine-native IMA4 load`** — Mac `.dat` files carry a distinct template-checksum family (`0x0003bcdf23c13061` vs PC `…33dc271f`) and lay SNDD out as a 16-byte record (Flags/Duration/DataSize/RawOffset), where the single PC layout the 32→64 bridge compiles is 72 bytes (`num_bytes`@56 / `data`@64). Walking Mac SNDD through the PC layout read garbage `num_bytes` → `SSiIMA_DecompressSoundData_Stereo` ran ~56 MB out of bounds → SIGSEGV on the first stereo sound (`mus_amasian1_1`, level 1). New per-file detector ([`TMtInstanceFile.isMac`](BungieFrameWork/BFW_Source/BFW_TemplateManager/BFW_TM_Game.c), set from the header checksum at load) + new [`BFW_TM_MacLayout.c`](BungieFrameWork/BFW_Source/BFW_TemplateManager/BFW_TM_MacLayout.c) that remaps the 16 Mac bytes into the shared `SStSoundData` (derives `nChannels` from Flags, forces `wFormatTag`≠2 to route to the Apple-IMA4 decoder not ffmpeg MS-ADPCM, copies the 8-byte container preamble so `TMrInstance_GetRawOffset` resolves the owning file). One branch at the bridge translate site selects it when `isMac`; PC files have `isMac==UUcFalse` and short-circuit byte-for-byte unchanged. Resolves §5 of the spec as a hand-written translate fn (not the swap-code-array approach — Mac→shared-struct is a remap with derived fields, which swap codes can't express).
- **`fix(audio): Mac IMA4 state words are big-endian — gated swap`** — with the crash gone the IMA4 decoder ran but produced uniform static across every Mac sound. Root cause: commit `5ca88b2` removed `UUmSwapBig_2Byte` from the IMA state-word reads (correct for PC little-endian `.raw`; its own message predicted *"will matter for any IMA-format sounds"*). Mac IMA4 stores each packet's predictor/step-index big-endian. Re-added the swap at the three decode sites in [`BFW_SS2_IMA.c`](BungieFrameWork/BFW_Source/BFW_SoundSystem2/BFW_SS2_IMA.c), gated on a new per-sound `SScSoundDataFlag_MacIMA4` set by the Mac translate — so PC decode is byte-identical (zero regression by construction).

User-verified: Mac level 1 loads from a save, menu music + in-level gunfire clean, no SIGSEGV; PC/CX data re-checked, unchanged. Stage 1 of the spec done (SNDD). A follow-up playtest (cloak/Daodan/explosions/combat/textures) + clean log scan then showed OSBD/BINA/TXMP already work through the shared layout — Stages 2–3 **de-scoped** (only SNDD needed a Mac-specific runtime path; they resolve bulk data via the already-wired `.sep` separate-file path). Cut release **1.3.1r1** (Mac + PC drop-and-play); [#37](https://github.com/andiyar/OniARM64/issues/37) closed.

### 2026-05-31 — Session 42: Native AVFoundation intro/outro cinematics (fixes #31)

- **`feat(cinematics): native AVFoundation playback for intro/outro .mov`** — the Bink FMV path (`BFW_Bink.c`) has been a dead no-op on ARM64 (`ONI_BINK_VIDEO=OFF` compiles every function body out, so `BKrMovie_Play` is a bare `return`), meaning the intro/outro never played. New [`Oni_Cinematic_macOS.mm`](OniProj/OniGameSource/Platform_SDL/Oni_Cinematic_macOS.mm) + C shim `ONrCinematic_PlayNative` plays the Feral-converted `intro.mov`/`outro.mov` (H.264 + LPCM, 512×384, already bundled in `Contents/Resources/`) via `AVPlayer`/`AVPlayerLayer` in a self-owned borderless window — hardware decode + audio for free. `ONrMovie_Play` ([Oni_Bink.c](OniProj/OniGameSource/Oni_Bink.c)) routes through it on Apple+SDL (`"intro.bik"` → `intro.mov`, extension stripped); the dead Bink body is preserved behind `#else` for Windows, and a missing asset is a graceful no-op so startup never blocks. The movie window is anchored to the Oni game window's monitor via `SDL_GetWindowWMInfo` (true fullscreen on the correct display for multi-monitor setups), letterboxed (`ResizeAspect`, no stretch), and skippable (Esc / any key / click). CMake gains an `if(APPLE)` block under the SDL branch: `enable_language(OBJCXX)`, links AVFoundation/CoreMedia/CoreVideo/QuartzCore/Cocoa, per-source `-fobjc-arc` + `SKIP_PRECOMPILE_HEADERS`. Bypasses RAD's Bink SDK entirely. User-verified (intro plays with audio on the correct monitor, Esc skips into the menu). Fixes [#31](https://github.com/andiyar/OniARM64/issues/31).

### 2026-05-28 — Session 39: Two gameplay crashes fixed — levels 2 and 3 now playable (fixes #32, addresses #33)

- **`fix(64bit): uintptr_t cast in New Game dialog level-list message`** — `ONiLevelList_GetLevelNumber` in `Oni_Dialogs.c:170` passed `(UUtUns32)(&level_name)` to `VMrView_SendMessage` (which accepts `uintptr_t`), truncating the stack address on 64-bit ARM64. Found via codebase-wide audit; not yet triggered in gameplay because level selection was done via cheats. Addresses [#34](https://github.com/andiyar/OniARM64/issues/34).
- **Codebase audit**: thorough sweep of all `(UUtUns32)` pointer-truncation patterns across the tree. Confirmed remaining sites are all in dead code: TOOL_VERSION editor (`Oni_Windows.c`, `Oni_Win_Tools.c`, `Oni_Win_Particle.c`), disabled features (`ONmUnitViewer`), PowerPC-only (`MS_Geom_Transform_AltiVec.c`), unused subsystems (networking, Bink). The `sizeof(UUtUns32)` stride pattern (session 34's `M3rDraw_State_Push` bug) has no remaining instances. Active gameplay code paths are clean after sessions 23–39's fixes.
- **`fix(materials): || vs && bounds-check bug in material/character validation`** — Bungie's original source used `||` instead of `&&` in three bounds-validation sites for unsigned values (particle impact tex_material, weapon impact tex_material, damage-owner character index). On 32-bit the always-true check was benign; on 64-bit the out-of-bounds `MAgMaterialArray` read returns NULL → SIGSEGV at `MArMaterialType_GetParent+32`. Triggered by the screamer particle weapon on level 2 (Deadly Brain fight). Also added runtime bounds checks in `MArMaterialType_GetParent`, `MArMaterialType_GetDepth`, and `MArImpactType_GetParent` as defense-in-depth. Fixes [#32](https://github.com/andiyar/OniARM64/issues/32).
- **`fix(sound): indirection table for TRtAnimation.soundName pointer truncation`** — `TRtAnimation.soundName` is `UUtUns32` in the TRAM template struct but holds a `char*` at runtime. On 64-bit the cast truncates the pointer → SIGSEGV in `AUrString_GetHashValue` when looking up impulse sounds during character death animation. Can't change the struct type without breaking the template bridge layout, so the `UUtUns32` slot now stores a 1-based index into a static `const char*` table. Triggered on level 3 (explosion + acid death). Addresses [#33](https://github.com/andiyar/OniARM64/issues/33) pending user verification.
- **Gameplay progress**: user played through the rest of level 2 (including Deadly Brain phase 3 with heavy particle effects) and most of level 3, confirming particle screamers, lasers, and environmental effects all work. Levels 1–3 now playable end-to-end.

### 2026-05-27 — Session 38: Widescreen portrait aspect-ratio fix (addresses #25)

- **`fix(render): uniform scaling for cinematic portraits at widescreen aspect`** — `OCiCinematic_Draw` ([Oni_Cinematics.c:260-314](OniProj/OniGameSource/Oni_Cinematics.c#L260)) scaled portrait sprites with independent X/Y factors from a 1024×768 virtual screen to actual resolution. ### 2026-05-26 — Session 37: HD bake pipeline — export-based index catches unnamed textures (#16)

- **`fix(pipeline): build instance index from OniSplit -export, not -list`** — the previous `build-hd-gamedata.sh` built its per-level instance index via `OniSplit -list`, which only returns named/public instances. Character body textures (`k4_head`, `KS_face` in some levels, `KO_` cop textures) are unnamed in several level .dat files — they exist in the data but have no name-table entry. The HD character mod 23951 (Konoko+Cops) targets these textures; the old index silently missed them, so mod overlays were never staged and Konoko rendered with vanilla textures (white-hair regression was likely a vanilla/mod texture mismatch artifact during partial overlay). Now: step 1 runs `OniSplit -export` per level upfront (cached — skips if export dir is newer than .dat), then builds the index by scanning exported filenames (`ls | sed` to decode `%2F` and strip `.oni`). This catches all instances OniSplit knows about, named and unnamed. The per-level bake loop reuses the cached export instead of re-exporting. Verified: `k4_head` now appears in the index for levels 0, 8, 11, 12, 13, 14, 18, 19 (was missing from 0, 8, 18 under `-list`). Addresses [#16](https://github.com/andiyar/OniARM64/issues/16) — pending user re-test of the full 15-mod bake with the new pipeline.

### 2026-05-26 — Session 37: Splash screen centering fix for oversized TXMBs

- **`fix(render): signed arithmetic for splash screen centering`** — `splash_screen_internal` ([Oni_GameState.c:602-603](OniProj/OniGameSource/Oni_GameState.c#L602)) computed centering offsets via `UUtUns16 x_location = (M3rDraw_GetWidth() - splash_screen->width) / 2`. When an HD texture mod produces a TXMB wider than the screen, the unsigned subtraction wraps to a large positive value (~64K), pushing the splash entirely offscreen. Changed to signed `int` arithmetic before converting to `float` for `point_screen.x/y`. Negative results (image wider than screen) now correctly crop the image rather than wrapping. The desktop-paint path (`WMiDesktop_Paint`) already uses signed types and was not affected.

### 2026-05-26 — Session 37: NULL-texture guard in BigBitmap renderer (addresses #28)

- **`fix(motoko): NULL guard in M3rDraw_BigBitmap and M3rDraw_Bitmap`** — `M3rDraw_BigBitmap` ([Motoko_Utility.c:654-656](BungieFrameWork/BFW_Source/BFW_Motoko/Manager/Motoko_Utility.c#L654)) walked `textures[index]` with no bounds check on `index < num_textures` and no NULL check on the texture pointer. When HD texture mods produce TXMB instances with sub-texture references that don't resolve (OniSplit repack can create TXMBs pointing at TXMPs not present in the .dat), the template system silently writes NULL into the pointer slot (`iBridgePrepare_ResolveTemplatePtr` returns success with resolved=NULL). `M3rDraw_Bitmap` then dereferences NULL at offset 0x84 (`width` field). Added: (1) bounds check `index >= num_textures → continue` in `M3rDraw_BigBitmap` (prevents out-of-bounds read when `num_x * num_y > num_textures`), (2) NULL check `texture == NULL → continue` (skips missing tiles), (3) belt-and-suspenders `inBitmap == NULL → return` at the top of `M3rDraw_Bitmap` since it's called from multiple sites. Bungie's original code is byte-identical here — the gap predates the port but only fires with modded data. Addresses [#28](https://github.com/andiyar/OniARM64/issues/28) pending user verification.

### 2026-05-24 — Session 36: Minimal ffmpeg build strips GPL deps from bundle (fixes #19)

- **13 dylibs → 3 dylibs.** Bundle's `Contents/Frameworks/` previously shipped SDL2 + 12 ffmpeg-related dylibs (libavcodec, libavutil, libswresample, libvpx, libdav1d, libmp3lame, libopus, libSvtAv1Enc, libssl, libcrypto, **libx264 (GPL-2+)**, **libx265 (GPL-2+)**) — all transitive deps of Homebrew's `--enable-gpl` ffmpeg bottle. The redistributed binary was effectively GPL-subject because of x264/x265 even though Oni never calls them at runtime. After this commit: SDL2 + a custom `libavcodec.62.dylib` (356 KB) + `libavutil.60.dylib` (720 KB), both LGPL-2.1, both linked only to Apple system frameworks. GPL exposure gone.
- **Approach:** Option C from #19 — rebuild ffmpeg ourselves with `--disable-gpl --disable-nonfree --disable-everything --enable-decoder=adpcm_ms` and a long list of explicit `--disable-*` flags (avdevice, avfilter, avformat, swresample, swscale, videotoolbox, audiotoolbox, coreimage, xlib, libxcb, sdl2, securetransport, network, protocols, devices, filters, muxers, demuxers, parsers, bsfs, encoders, iconv, zlib, lzma, bzlib). The only enabled component is the ADPCM_MS audio decoder — the one thing Oni actually uses (`BFW_SS2_Platform_OpenAL.c:307-411`). Pinned to ffmpeg 8.1.1, matching the Homebrew bottle ABI Oni was tested with — same libavcodec.62 SONAME, same decoder code, zero risk to the audio path that's been historically fragile (sessions 23 ap_wiz, 27-29 gunfire-lag, 29 OpenAL buffer cache, 30 Daodan ap_wiz backstop).
- **Why not the more aggressive Option A/B** (replace ADPCM decode with AudioToolbox / hand-rolled): audio path's track record bites. Touching the working decoder for marginal cleanliness gain wasn't worth the regression risk. Option C is the conservative "drop the GPL libs without touching audio code" path.
- **Files:**
  - `scripts/build-ffmpeg.sh` — new. Downloads pinned ffmpeg 8.1.1 source, configures minimal, builds, installs to `extern/ffmpeg/`. Idempotent. Cached tarball at `extern/build/ffmpeg-8.1.1.tar.xz` on subsequent runs.
  - `OniProj/OniCMakeProjs/OniProj/CMakeLists.txt` — prepends `extern/ffmpeg/lib/pkgconfig` to `PKG_CONFIG_PATH` if it exists; falls back to Homebrew otherwise (preserves the dev workflow where `extern/ffmpeg/` doesn't exist yet).
  - `macos/build-bundle.sh` — BFS-walk awk pattern now matches both `/opt/homebrew/` AND `$SOURCE_DIR/extern/`. The walk picks up our custom dylibs cleanly.
  - `.gitignore` — `/extern/` (the build output, not committed).
  - `README.md` — Bundled third-party software section trimmed from 10-row GPL-flagged table to 2 rows (SDL2, FFmpeg minimal). GPL warning gone.
  - `THIRD_PARTY_LICENSES.md` — dropped sections for libmp3lame, libopus, libvpx, libdav1d, libSvtAv1Enc, OpenSSL, libx264, libx265. Replaced GPL note with a "No GPL components" note pointing at the build script.
- **Verification:** clean cmake-from-scratch + make + make oni_app → bundle ships with exactly `libSDL2-2.0.0.dylib` + `libavcodec.62.dylib` + `libavutil.60.dylib`. Total dylib payload 2.5 MB (was ~50 MB+). Smoke-test launch: PID alive 4s, no SIGSEGV. User ear-test on tutorial level: music + gunfire + combat impacts + voice + footsteps all audible and correct via the new minimal libavcodec. Audio cache stats from prior runs corroborate: ADPCM decode produces non-zero hit rates, which requires successful decodes. `fixes #19`.
- **Bundle weight win is real but not headline.** The 95 MB bundle total is now dominated by the intro/outro `.mov` cinematics (89 MB combined). The dylib win is ~50 MB → 2.5 MB. Removing the `.mov` AVFoundation gap (separate issue, not filed yet — would require wiring intro/outro through AVFoundation since they bypass BFW_Bink.c entirely) would cut another 89 MB and leave a ~6 MB total bundle.

### 2026-05-24 — Session 34: Level-3 (bio lab) cinematic crash — draw-state push truncated 8-byte pointers to 4 (#29)

- **Symptom:** consistent SIGSEGV in `gl_quad` (NULL deref at `gl_geometry_draw_method.c:222`) during the bio-lab opening cinematic, the moment Barabas fires his energy projectile (`benrgy_fly`). User crashed ~10 times today across multiple `Oni-*.ips` reports. Stack: `gl_quad ← M3iSort_Draw_Quad ← M3rSort_Frame_End ← M3rDraw_Commit_Alpha ← ONiGameState_Display_Overlay_Elements`. Last log line before crash: `[SS2] SSrAmbient_Start name='benrgy_fly' usable=1 pos=0x16b696cb4`.
- **Root cause via disasm of the crashing instruction** (`gl_quad + 0x7ac`): `ldp s0, s1, [x8]` where `x8 = state_ptr[3] + indices[0]*8 = NULL + 0`. `state_ptr[3]` is `M3cDrawStatePtrType_EnvTextureCoordArray`. The crash path is `gl_quad`'s env-map branch, which we reached because `state_ptr[6] = EnvTextureMap` read non-NULL, which it shouldn't have.
- **The bug ([Motoko_State_Draw.c:270](BungieFrameWork/BFW_Source/BFW_Motoko/Manager/Motoko_State_Draw.c#L270)):** `M3rDraw_State_Push` copied the pointer stack with stride `sizeof(UUtUns32) * NumTypes` = 32 bytes. On 64-bit, the pointer stack is `void**` — 8 pointers × 8 bytes = 64 bytes. Only the first 4 pointers (entries 0-3: `ScreenPointArray`, `ScreenShadeArray_DC`, `TextureCoordArray`, `EnvTextureCoordArray`) actually got copied; entries 4-7 (`BaseTextureMap`, `LightTextureMap`, **`EnvTextureMap`**, `VertexBitVector`) retained whatever stale values were left in the new stack slot from a previous unrelated push. So during the alpha-quad pass, `state_ptr[6]` held a stale-but-still-valid env-map pointer from an earlier env-mapped scope, `gl_private_state_update` ([gl_engine.c:441](BungieFrameWork/BFW_Source/BFW_Motoko/Engines/DrawEngine/OpenGL/gl_engine.c#L441)) saw it as non-NULL and switched `geom_draw_mode` to env-map, and `gl_quad` then read `state_ptr[3]` — which the alpha quad never set — and dereferenced NULL.
- **Why level 3 specifically:** bio lab has shiny env-mapped lab surfaces (heavier env-map usage than levels 1/2), so by the time the alpha pass runs there's a strong chance entry 6 is a stale-but-valid pointer from prior env render. Barabas's beam particle was just the first alpha submission of that frame to read it. Issue #7 (level-2 cinematic crash — single observation, no repro) plausibly shares the root cause.
- **Fix (`fix(motoko): copy 8-byte pointers in draw-state push, not 4-byte`):** one-line change — replace `sizeof(UUtUns32)` with `sizeof(void*)` so the push copies all 64 bytes of the pointer array. After the fix, entry 6 is correctly copied as NULL from the outer scope, the env-map branch isn't taken, and the alpha quad renders via `_geom_draw_mode_default` using only entries 0/1/2 (which `M3iSort_Draw_Quad` does set). Addresses [#29](https://github.com/andiyar/OniARM64/issues/29) pending user verification of the bio-lab repro.
- **Framing**: port-exposed 32→64. Bungie source is byte-identical here (`diff -w` empty); the 2000-12-21 code was correct for its 32-bit target where `sizeof(void*) == sizeof(UUtUns32) == 4`. Same family as issue #11 (`AUrQSort_32` treating pointer arrays as `UUtUns32`).

### 2026-05-23 — Session 29: Gunfire-lag root cause + OpenAL buffer cache

- **Gunfire lag root-caused via external `sample` profiling.** After five fflush-gating passes in sessions 27-28, user reported the lag still dramatic on gunfire and worse under strafe + Daodan. Three macOS `sample` captures (`/tmp/oni-sample-20260523-{113844,114424,125918}.txt`) revealed the smoking gun: `SS2rPlatform_SoundChannel_SetSoundData → alBufferData → OALBuffer::AddAudioData → usleep → __semwait_signal`. Apple's OpenAL implementation internally calls `usleep` per PCM upload, blocking the main thread ~1ms per `alBufferData` call. Under stress (cheat + Daodan + strafe-fire), **4951 main-thread samples sat in this single syscall path = 8.3% of wall clock**. Issue #9's stated mechanism (ADPCM decode + alGetError sync) was refuted by the same profile: alGetError = 0 samples, libavcodec = ~13 samples total — neither is hot. The cost is `alBufferData` itself, not the decode preceding it.

First play of a sound: `alGenBuffers` + `alBufferData` (eats the usleep once). Subsequent plays: just `alSourcei(source, AL_BUFFER, cached_id)` — no PCM upload, no usleep. Channel's per-channel `pd.buffer` is preserved as fallback for `alGenBuffers` failure and as the legacy path when cache is disabled. Cache freed in `SS2rPlatform_Terminate` after dumping hit/miss stats. Env-var escape hatch `ONI_AUDIO_CACHE=0` reverts to per-call upload without a rebuild.
- **Post-fix verification (clip 3, same stress scenario, 90s `-wait` sample):** **0 `alBufferData` samples** anywhere in the profile (down from 4951). `SS2rPlatform_SoundChannel_SetSoundData` dropped from thousands of leaf samples to 17 — not because it's called less, but because each call now returns in microseconds and the 1ms sampler rarely catches it in flight. `OSrImpulse_Play` correspondingly dropped from 4491 to 243. ### 2026-05-20 — Session 25: Pursuit movement FIXED — embedded PHtRoomData bridge alignment

- **The fix landed and is user-verified.** Konoko stood in front of an NPC in level 1; the NPC walked over and engaged. - **Root cause (`fix(64bit): bridge embedded PHtRoomData alignment in AKVA template`, commit `6c030e0`):** the 32→64 template-instance bridge walker in `BFW_TM_Bridge.c` (which translates on-disk 32-bit instance data into 64-bit in-memory C structs) walks the swap-code stream as a flat sequence of scalars/pointers/arrays. There is no swap-code marker for "begin embedded `tm_struct`" — those boundaries were a no-op on 32-bit because no `tm_struct` had an alignment requirement larger than 4. On 64-bit, `PHtRoomData` (embedded inside `AKtBNVNode`) contains pointers (`compressed_grid`, `debug_info`) and therefore has alignment 8. The C compiler 8-aligns the embedded struct by inserting 4 bytes of pad before `roomData` in `AKtBNVNode`. The walker, oblivious, placed `gridX` and `gridY` at dst offsets 28/32 — the C compiler has them at 32/36. Because the walker correctly aligns `RawPtr` to 8 when it later hits `compressed_grid`, the drift cancels at offset 40 and every field after `compressed_grid` ends up correct. So the visible damage is exactly: `room->gridX` reads disk's `gridY`, `room->gridY` reads memset-zero.
- **Why the symptom looked like AI movement and not the template manager:** `room->gridY = 0` made every `(inY >= room->gridY)` bounds check in `ASiDetermineEndPoint` (Oni_AStar.c:1795) succeed for all `inY ≥ 0`, so no destination square was ever passable. `ASrPath_Generate` returned `UUcFalse` for every call. Inside `AI2iMovementState_SetupPath`, that triggered `AI2_ERROR` + `AI2rMovementState_ClearPath` — but `AI2_ERROR_REPORT = 0` in release, so the error was completely silent. The visible symptom was just "NPC sees you, enters Pursuit, never moves." Session 24's diagnostics narrowed to "grid-path generation is empty" but couldn't see why. Session 25 added three GRID-DBG tracers, the very first SetParams event printed `gridXY=(80x0)`, and the bridge bug became immediately obvious.
- **Fix scope:** a small post-pass (`iFixupEmbeddedStructAlignment`) in `TMrBridge_BuildDescriptor` recognises the `'AKVA'` template tag and shifts the two leading scalars in the element sub-descriptor by +4 bytes each. 64 lines added, all in `BFW_TM_Bridge.c`. No changes to swap codes, no checksum changes, no on-disk format changes.
- **Audit deferred:** `PHtRoomData` is the only known embedded multi-pointer `tm_struct` in this codebase (grep confirms it's embedded only in `AKtBNVNode` per `BFW_Akira.h:516`). A systematic audit of the other 113 templates for similar patterns is a follow-up — any template embedding a `tm_struct` whose alignment is `> 4` on 64-bit will have an equivalent bug, manifesting as some other "silent struct field is zero" symptom downstream.
